# 前台编辑页面产品建议

## 一、目标定义

基于当前自动化脚本能力，这个“前台编辑页面”不应该只是一个简单表单页，而应该定位为：

**自动打卡任务配置与运行控制台**

页面目标不是单纯编辑配置，而是让用户可以完成以下事情：

- 配置自动化任务参数
- 查看当前运行状态
- 管理随机打卡时间窗口
- 执行环境自检
- 快速排查异常
- 控制任务启停与调试

## 二、为什么需要这个页面

当前脚本已经具备以下能力：

- 定时调度 morning / evening 两个随机执行窗口
- 调度状态持久化
- 工作日联网判断
- 设备状态检查
- `status / schedule / doctor / run / debug` 命令支持
- 成功/失败通知
- `scrcpy` 调试观察能力

如果前台页面只支持“编辑参数”，用户仍然需要回到命令行查看运行状态、检查设备、确认排期和排查异常，产品闭环是不完整的。

因此这个页面应该承担“配置 + 监控 + 操作 + 排障”的一体化能力。

## 三、建议新增的核心功能

### 1. 基础配置编辑

建议支持以下字段的可视化编辑：

- 设备序列号 `serial`
- 应用包名 `package`
- 应用名称 `app_label`
- 启动后停留时长 `delay_after_launch`
- 轮询间隔 `poll_interval`
- `scrcpy` 观察模式开关
- 成功通知开关
- 工作日校验开关
- 工作日接口地址
- 工作日接口超时时间
- 状态文件路径 `state_file`

这一部分对应当前脚本的主要启动参数，属于页面的基础能力。

### 2. 打卡时间窗口编辑

建议把 morning / evening 的随机窗口前台化，支持：

- 上午打卡窗口开始时间
- 上午打卡窗口结束时间
- 下午打卡窗口开始时间
- 下午打卡窗口结束时间
- 清晰提示“系统会在时间窗口内随机选择一个时刻执行”
- 展示当前已抽取的今日执行时间
- 支持“重新抽取今日计划时间”
- 展示最近一次完成日期

这是页面中最核心的产品能力之一，因为当前脚本中的时间窗口是固定写死的，前台最有价值的编辑项就是这部分。

### 3. 运行状态面板

建议增加一个状态总览区域，展示：

- 当前任务状态：运行中 / 已停止 / 异常
- 当前设备状态：已连接、未授权、已断开
- 当前是否为工作日
- `scrcpy` 状态：运行中 / 未运行
- 上午下一次执行时间
- 下午下一次执行时间
- 最近一次成功执行时间
- 最近一次工作日校验结果

这部分对应现有脚本中的 `status` 与 `schedule` 能力，适合做成首页信息卡片。

### 4. 一键运维能力

建议在页面提供直接操作入口：

- 一键自检，对应 `doctor`
- 一键查看当前排期，对应 `schedule`
- 一键刷新设备状态
- 一键启动任务
- 一键停止任务
- 一键切换到调试模式
- 一键试运行一次自动化动作

如果没有这部分，用户即使完成编辑，也无法在页面里验证配置是否真正生效。

### 5. 日志与执行记录

建议增加执行记录和日志查看能力：

- 最近执行日志列表
- 成功/失败状态
- 失败原因摘要
- 工作日接口请求结果
- 设备授权提醒记录
- 每日执行时间线

相比只暴露原始日志文件，这种结构化展示更适合运营和排障。

### 6. 异常提醒与校验保护

建议在页面层增加必要的提示与保护：

- 设备未连接提醒
- adb 未授权提醒
- 工作日接口异常提示
- 保存前表单校验
- 时间窗口合法性校验
- 超时与轮询参数下限校验
- 关键配置变更后的二次确认

## 四、产品优先级建议

### MVP 必须具备

- 基础配置编辑
- 时间窗口编辑
- 当前状态展示
- 一键自检
- 日志查看
- 启动/停止控制

### 第二阶段建议增加

- 调试模式管理
- 历史统计报表
- 多设备支持
- 多任务模板
- 提示文案编辑

## 五、页面定位建议

这个页面不建议命名为“编辑页”，更建议命名为：

- 自动打卡任务配置台
- 自动化任务控制台
- 打卡任务配置与运行中心

其中最推荐的名称是：

**自动打卡任务配置与运行控制台**

## 六、结论

从产品设计角度，这个页面要解决的不是“参数编辑”单点问题，而是把当前脚本已经具备的能力前台化，形成完整的操作闭环：

- 可配置
- 可查看
- 可执行
- 可排障
